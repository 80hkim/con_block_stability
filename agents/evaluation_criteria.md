# Evaluation Criteria — 공용 채점 기준

이 문서는 Evaluator, Numerical Verifier, AI Prompt Auditor가 공통으로 참조하는 채점 기준이다.
각 에이전트는 자신의 담당 영역에 해당하는 기준만 적용한다.

---

## 1. 설계기준 정합성 (Evaluator 담당)

### 1.1 안전율 기준값
| 검토 항목 | 상시 | 지진시 | 근거 |
|-----------|------|--------|------|
| 활동 (Joint Sliding) | Fs ≥ 1.2 | Fs ≥ 1.1 | KDS 64 55 20, 4.1.3.2 |
| 전도 (Overturning) | Fs ≥ 1.2 | Fs ≥ 1.1 | KDS 64 55 20, 4.1.3.3 |
| 직선활동 (Base Sliding) | Fs ≥ 1.2 | Fs ≥ 1.1 | KDS 64 55 20 |

**FAIL 조건**: 코드의 안전율 상수가 위 값과 다른 경우

### 1.2 지지력 검토 방식
- 종래식 하중분산법 2단계 구조여야 한다
  - 1단계: 사석 마운드 상면 `p1 ≤ qta`
  - 2단계: 기초지반 상면 `p1' ≤ qa`
- 최종 판정: 두 조건 모두 만족 시 O.K

**FAIL 조건**: `qmax ≤ qa` 단일 조건으로 판정하는 경우

### 1.3 하중조합 정의
- 견인력 미적용(8조합): 상시 비계류 4 + 지진시 4
- 견인력 적용(12조합): 상시 비계류 4 + 상시 계류 4 + 지진시 4
- 계류 조합에만 견인력 포함, 지진시에는 견인력 제외 (KDS 64 55 20 §4.1 [참고](11))
- 잔류수압은 모든 하중조합에 적용

**FAIL 조건**: 하중조합의 그룹(비계류/계류/지진) 분류 또는 견인력 적용 조건이 변경된 경우

### 1.4 원호활동
- 이 앱에서 직접 계산하지 않는다
- "별도 프로그램 수행" 안내가 유지되어야 한다

**FAIL 조건**: 원호활동 직접 계산 로직이 추가된 경우

## 2. 수치 정확성 (Numerical Verifier 담당)

### 2.1 토압 계수
| 수식 | 검증 방법 |
|------|----------|
| 랜킨 Ka = tan²(45° - φ/2) | φ=30° → 0.3333 |
| 쿨롱 Ka | φ=30°, δ=15° → 교재 기준값 대조 |
| M-O Kea | kh=0일 때 쿨롱 Ka와 동일해야 함 |

### 2.2 단위 체계
| 항목 | 기대 단위 |
|------|----------|
| 힘 | kN/m (단위폭당) |
| 압력/응력 | kPa (= kN/m²) |
| 길이 | m |
| 단위중량 | kN/m³ |
| 각도 | ° (도) → 내부 계산 시 rad 변환 |

**FAIL 조건**: 단위 혼용 (예: t, tf, kgf 등 SI 외 단위 사용) — 이 프로젝트는 kN, kN/m², kN/m³, m 만 사용

### 2.3 허용 오차
- 중간 계산값: ±0.001
- 안전율 최종값: ±0.01
- 응력/압력: ±0.1 kPa

### 2.4 경계값 동작
| 입력 | 기대 동작 |
|------|----------|
| φ = 0° | Ka = 1.0 |
| kh = 0 | 지진 관련 항 = 0, Kea = Ka(쿨롱) |
| 블록 1단 | 정상 계산 (에러 없음) |
| 수위 = 해저면 | 수압 = 0 |

## 3. AI 프롬프트 안전성 (AI Prompt Auditor 담당)

### 3.1 필수 금지 규칙
| ID | 금지 사항 | 위험도 |
|----|----------|--------|
| P1 | `FS_br = qa/qmax` 등 임의 안전율 식 생성 | HIGH |
| P2 | 원호활동 값 추정/창작 | HIGH |
| P3 | 결과에 없는 기준값 추측 | HIGH |
| P4 | 관성력 증가를 안정성 보강 효과로 서술 | MED |
| P5 | 하중조합 그룹(비계류/계류/지진) 오인 | MED |

**FAIL 조건**: 위 금지 규칙이 systemPrompt에서 누락된 경우

### 3.2 응답 포맷 호환성
- NG 분석: 마크다운 텍스트 (파싱 불필요)
- 최적화: JSON 파싱 가능한 구조 (`optimizedData` 호환)
- CTX_TEXT / CTX_JSON: 올바른 구분자로 전달

**FAIL 조건**: 응답 포맷 변경으로 기존 파싱 로직이 깨지는 경우

## 4. 코드 품질 (Evaluator 담당)

### 4.1 스타일 일관성
- 기존 IIFE 패턴 (`const X = (function() { ... })()`) 유지
- 들여쓰기: 4스페이스
- 한글 주석 허용 (기존 관례)
- camelCase 변수명

### 4.2 회귀 위험도 분류
| 위험도 | 기준 |
|--------|------|
| LOW | UI 텍스트/스타일만 변경 |
| MED | 입력/출력 인터페이스 변경 |
| HIGH | 계산 로직, state 구조, API 프롬프트 변경 |

### 4.3 금지 패턴
- `eval()`, `innerHTML` (XSS 위험이 있는 사용자 입력 직접 삽입)
- 전역 변수 추가 (기존 IIFE/클래스 패턴 밖)
- `console.log` 잔존 (디버깅용)
